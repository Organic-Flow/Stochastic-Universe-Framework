Experiment 01

Network Partition
Split-Brain Routing Without Manual Failover

The Claim

MorphoCloud automatically detects network partitions and reroutes through an alternative relay node — with zero manual failover, zero configuration change, and zero dropped requests.

The Problem with Classical Approaches

In distributed systems, a network partition — a firewall rule, NIC failure, or routing loop — causes split-brain: nodes cannot communicate and the system either stalls or issues conflicting responses. Resolution requires manual failover, an external health checker, or a consensus protocol (Raft/Paxos) demanding a quorum. None of these happen automatically at the application layer.

Experiment Setup
01

Three-Node Cluster

MASTER, node-B, node-C — a FinTech pipeline distributed across nodes. Normal operation: MASTER routes to node-B for processing.

02

Partition Injection

A firewall rule severs the MASTER → node-B connection mid-operation. Five stimuli follow with node-B unreachable.

03

Baseline

Without organic rerouting, all 5 requests fail. No path exists in the static routing table for the partition scenario.

Topological Rerouting (Partition Healed)

Swarm Intelligence: When the direct MASTER → Node B path is severed (Partitioned), the field automatically discovers Node C as a valid relay, routing traffic through the alternative path without any packet loss.

Technical Due Diligence
Q

Doesn't BGP failover or an external load balancer solve this?

A

BGP and load balancers operate at the network layer and require pre-configuration. MorphoCloud solves this at the application layer: the cascade itself dynamically discovers that a node is unreachable and uses the global registry to find an alternative path — no routing tables, no BGP peers, no configuration update required.

Q

How do you avoid split-brain without Raft/Paxos consensus?

A

We use Eventual Topology Convergence rather than strong consistency. If the cluster splits, each partition continues operating locally (graceful degradation). When the partition heals, both sides resync via the global registry. For applications requiring strict consistency (e.g. bank balance), MemoryCell enforces a lock. For most microservices, availability takes precedence over consistency — and this is the right tradeoff.

Experiment 02

External Dependency
PersistentCell Graceful Degradation

The Claim

When an external API fails, MorphoCloud continues serving stale data with an automatic stale: True flag — instead of crashing the cascade. Graceful degradation requires zero fallback code.

The Problem with Classical Approaches

In classical systems, if an external API — a price feed, database, or third-party service — fails, the request throws an exception. Graceful degradation requires additional code: try/except blocks, a cache layer, a circuit breaker, or a custom fallback strategy. All of it is manually implemented, all of it is a separate concern from the business logic.

Experiment Setup

Pipeline

StockTickerCell (PersistentCell — polls a fake price feed every 3s) → PriceAlertCell (TransientCell — reads the signal downstream).

Phase 1: Normal operation — 5 stimuli, feed live. Phase 2: Feed killed → poll failures accumulate → cell switches to stale-serving mode automatically.

03

Baseline

Without PersistentCell semantics, every feed failure causes a cascade crash. Downstream cells receive no data and the pipeline halts.

Technical Due Diligence
Q

Isn't this just a cache layer?

A

A cache layer stores the last value but has no awareness of staleness — that logic must be written manually. PersistentCell automatically emits the stale: True flag, degrades its reliability cumulatively, and issues SOS alerts. The downstream consumer is informed without a single line of additional code.

Experiment 03

Poisoned Pill
Reputation System & Automatic Quarantine

The Claim

MorphoCloud identifies and isolates a malicious cell through blame back-propagation — automatically, with no manual audit, no whitelist, and no human intervention. In a live FinTech pipeline, the poisoned fraud detector is quarantined at transaction #31.

The Problem with Classical Approaches

In classical microservice systems, if a service returns incorrect results — a fraud detector that always approves, a pricing engine that always discounts — there is no automatic isolation mechanism. Detection happens through logging and monitoring, the decision is made by humans, and isolation requires a manual deployment change. The mean time to detect is 20–60 minutes. The mean time to fix is longer.

Experiment Setup
01

FinTech Pipeline

authenticate_user → check_fraud → process_transaction → send_receipt. Four cells in sequence.

02

The Poison

check_fraud is poisoned: it always returns CLEARED, regardless of the transaction. process_transaction is wise: it detects suspicious approvals and raises blame signals against check_fraud.

03

100 Transactions

68 normal, 32 suspicious. Each suspicious transaction triggers a blame penalty of −0.05 on check_fraud. Quarantine threshold: reputation < 0.15.

The reputation of check_fraud degraded cumulatively through blame back-propagation. At transaction #31, quarantine triggered automatically — the poisoned cell was isolated from the ecosystem.

Reputation & Quarantine Cycle

Auto-Immune Response: As the Poisoned Fraud Cell (New/Untrusted) issues suspicious clearances, its reputation collapses. Once it crosses the 0.15 threshold, the organism surgically quarantines it from the financial pipeline.

Technical Due Diligence
Q

Can a malicious cell "hide" by making only occasional mistakes?

A

Yes — and this is exactly what is demonstrated. check_fraud starts with reputation=0.50 (already in the quarantine zone). The blame penalty (−0.05) exceeds the reward (+0.01). A cell making fewer mistakes would take more transactions to quarantine — but the outcome is the same. The convergence speed is proportional to how often the cell errs.

def _get_reputation(cloud: MorphoCloud, cluster_id: str, cell_name: str) -> float:
    status = cloud.engine.get_cluster_status(cluster_id)

    return min(c["traits"].get("reputation", -1.0) for c in cells)
